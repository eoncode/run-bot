## What this project is

A macOS menu bar app called **RunnerBar** that monitors GitHub Actions self-hosted runners.
It lives in the menu bar as a status dot whose colour reflects aggregate runner state. Clicking
it opens an NSPopover panel listing runners, their live status, active jobs, and workflow
action groups. It also **manages** local runners: add/install, start/stop lifecycle via
`launchctl`, edit config, view step logs, and run per-scope failure hooks.

## Hard constraints

- **No Xcode project checked in by hand** — the `.xcodeproj` is generated by **XcodeGen** from
  `project.yml`. Do not hand-edit or commit `.xcodeproj` / `.xcworkspace` / `.xib` / storyboards.
- **SwiftPM is the source of truth** — `Package.swift` defines the targets; `project.yml` mirrors
  it for the Xcode/UI-test path.
- **Swift 6.2**, strict concurrency. The `NonisolatedNonsendingByDefault` upcoming feature is
  enabled on every target.
- **macOS 26 (Tahoe)** minimum deployment target. Liquid Glass aesthetic is expected; you may use
  macOS 26 APIs freely.
- **Apple Silicon (arm64) only.** `build.sh` pins `--arch arm64` and an explicit build path on
  purpose — do **not** revert to an arch-neutral path (it caused stale-artifact debugging pain).
- **Dependencies:** only `apple/swift-collections` is allowed (used for `Deque`/ordered types in
  the poll loop). Do not add third-party dependencies without asking — prefer stdlib and system
  frameworks.
- **No Interface Builder** — all UI is programmatic AppKit + SwiftUI.

## Targets

```
RunnerBarCore   (library)      — pure logic, no UI; the testable core
RunnerBar       (executable)   — AppKit/SwiftUI app; depends on RunnerBarCore
RunnerBarCoreTests (test)       — swift-testing suite for the core
```

**Core must never import the app target.** All app-layer dependencies are injected into Core via
protocols and closures (see `RunnerPollerProtocol`, `RunnerViewModelProtocol`, the various
`*StoreProtocol`s). This boundary is load-bearing — keep it.

## Architecture & concurrency (read before touching data flow)

- **Actor-based.** `RunnerPoller` (Core actor) owns the GitHub poll loop and all derived state.
  `LocalRunnerStore`, `RunnerProxyStore`, `ScopePreferencesStore` are actors too (actor-per-concern).
- **Data flow is `@Observable`, not Combine.** `RunnerPoller` writes results into an injected
  `@MainActor @Observable RunnerState` via `await MainActor.run { … }`; SwiftUI observes it
  directly. There is no `PassthroughSubject` and no `RunnerViewModel` push-coupling anymore.
- **Immutable value models.** `RunnerModel`, `Runner`, and config types are fully `Sendable` with
  `let` properties; mutate via `copying(…)`. No `@unchecked Sendable` in production types except
  the one documented sign-off on `PollLoopCoordinator`.
- **Structured concurrency for timers.** Poll/observation tasks are owned by `PollLoopCoordinator`.
- **Atomic snapshots** are used to avoid TOCTOU races (e.g. rate-limit state).
- Full details live in `docs/architecture/data-model.md`, `concurrency-overview.md`,
  `swift-concurrency-lexicon.md`, and the canonical list in `docs/principles/project-principles.md`.

## GitHub networking

**Do not shell out to `gh api`.** Networking goes through a real `URLSession`-based transport:

- `GitHubTransportProtocol` — the abstraction; `GitHubURLSessionTransport` (`struct GitHubTransport`)
  is the concrete implementation.
- Module-level `ghAPI` / `ghAPIPaginated` shims route requests through the active transport.
- Requests are built by `GitHubRequestBuilder`, decoded by `GitHubResponseDecoder` into typed
  `Codable` models, and rate-limited via the `GitHubRateLimitHandler` actor.
- Pagination uses GitHub Link headers (`ghAPIPaginated`).

For tests, inject a stub conforming to `GitHubTransportProtocol` (see `GitHubTransportShimTests`
and the `StubURLProtocol` in the paginated tests) — never hit the live network in tests.

## Auth

Never prompt for a token interactively in code paths other than the Settings OAuth flow. Tokens are
resolved by `GitHubTokenCache` / `OAuthService`. Supported sources, in priority order:

1. In-app **OAuth sign-in** (Settings panel) — token stored in Keychain
2. `GH_TOKEN` environment variable
3. `GITHUB_TOKEN` environment variable
4. If all fail: surface
   `"Sign in via Settings or set GH_TOKEN / GITHUB_TOKEN env var, then try again."`

> ⚠️ `gh auth login` / `gh auth token` is **not** a supported auth path (removed in Batch 18,
> `GitHubTokenCache.swift`). Do not reference it in user-facing strings, comments, or instructions.

## UI rules

- No Dock icon (`LSUIElement` agent / menu-bar-only app).
- Menu bar dot reflects aggregate runner state (green / yellow / gray) derived by `AggregateStatus`.
- Click → NSPopover panel (`PanelMainView`). **Popover side-jump is a known hazard** — before
  touching panel sizing/anchoring, read `docs/ui/popover-side-jump-prevention.md`,
  `docs/ui/nspopover-dynamic-width.md`, and respect the regression guards in `PanelMainView` /
  `PanelVisibilityState` (refs #375–377).
- Sheets/dismiss behaviour: see `docs/ui/nspopover-dismiss-and-sheets.md`.
- Follow the macOS 26 Liquid Glass design system (`DesignTokens`, `PanelViewModifiers`); support
  dark & light mode (`docs/design/dark-light-mode-support.md`).
- Scope config (watched repos/orgs) persists in `UserDefaults` via `ScopeStore`.

## In scope (this app DOES do these now)

The original "out of scope" list is obsolete. RunnerBar now supports:

- Adding / installing new self-hosted runners (`AddRunnerSheet`, install flow)
- Start / stop / remove runner lifecycle via **`launchctl`** (`RunnerLifecycleService`)
- Local runner discovery by scanning LaunchAgent plists (`RunnerModelParser`, `LocalRunnerStore`)
- Live CPU/memory metrics for busy runners
- Workflow action groups, active jobs, and **step log viewing** (`LogFetcher`, `StepLogView`)
- Per-scope **failure hooks** (`FailureHookRunnerUseCase`)
- Notifications and launch-at-login (`NotificationPreferences`, `LoginItem`)
- Multi-scope monitoring (multiple repos and orgs at once)
